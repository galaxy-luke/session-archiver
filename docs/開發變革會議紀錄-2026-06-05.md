# DAD 系統開發變革會議紀錄

**日期：** 2026-06-05  
**參與人員：** 開發團隊、產品負責人  
**會議主題：** DAD 數位廣告數據平台 — 重大架構變革報告

---

## 一、變革背景

系統初期以「用戶導向」設計，所有資源（資料來源、映射模板、輸出報表）直接掛在用戶底下。但在與實際使用者進行深度討論後，發現真實的業務場景是：

> **每個廣告客戶（品牌/產品線）是一個獨立專案，團隊成員在不同專案中扮演不同角色。**

因此決定將系統方向調整為「**專案導向（Project-based Tenancy）**」，以專案為核心單位組織所有數據與權限。

---

## 二、重大變更項目

### 1. 系統架構：從「用戶導向」改為「專案導向」

| 面向 | 變更前（用戶導向） | 變更後（專案導向） |
|------|-------------------|-------------------|
| 核心單位 | 以使用者為中心 | 以專案為中心 |
| 資料歸屬 | 資源直接掛在用戶下 | 所有資源歸屬於專案 |
| 權限模型 | 全域角色（管理員/一般用戶） | 專案級角色（專案管理員/編輯者/檢視者） |
| 數據隔離 | 所有用戶看到相同數據 | 每個專案的數據完全隔離 |
| 多租戶 | 無 | Filament Tenant 多租戶架構 |

**變更原因：** 使用者反映實際工作場景中，一位數據分析師可能同時負責多個廣告客戶，需要在不同專案間切換，且各專案的數據必須隔離。原有的用戶導向設計無法滿足此需求。

**影響範圍：** 全部 12 個 Filament Resource 重寫，6 個 Model 關聯重新設計。

---

### 2. 技術框架升級：Filament 5.x 全面遷移

| 面向 | 說明 |
|------|------|
| 版本 | 從 Filament 3.x 升級至 5.x |
| 原因 | 支援多租戶（Tenant）功能、新版 Schema 架構、效能優化 |
| 變更內容 | 全部 Resource、Table、Form、Page 重寫 |
| 影響文件數 | 65+ 個 PHP 檔案 |

**主要技術改動：**
- Action 命名空間統一：`Filament\Tables\Actions\*` → `Filament\Actions\*`
- 表格方法更名：`recordActions()` → `actions()`
- Schema 架構取代舊版 Form/Table 分離設計
- Tenant 多租戶配置：以 Project 模型作為 Tenant

---

### 3. 權限系統重構

| 面向 | 說明 |
|------|------|
| 套件 | 引入 `spatie/laravel-permission` |
| 全域角色 | `super_admin`（系統管理）、`admin`（一般管理） |
| 專案角色 | `project_admin`（專案管理員）、`project_editor`（編輯者）、`project_viewer`（檢視者） |
| 權限隔離 | 使用者、角色、權限管理僅限 super_admin 存取 |

**設計考量：** 廣告代理商的組織結構中，系統管理員負責平台維運，專案管理員負責個別客戶的數據管理。兩者權限範圍完全不同，需要分層設計。

---

### 4. 數據分析功能強化

| 功能 | 說明 |
|------|------|
| 分析樣板 | 支援每週/每月/廣告活動/素材/自訂欄位分組 |
| 每週起始日 | 可配置（週日~週六），不再硬編碼 |
| 第一週處理 | 可選擇「從資料起始日」或「從完整週起始日」（不足天數顯示 0） |
| WoW 比較 | 自動計算最後兩週的環比變化率 |
| 公式欄位 | 支援 CTR、ROAS 等衍生指標的公式重算 |
| 分享樣板庫 | 分析樣板可快照發布為全域分享模板，跨專案複用 |

---

### 5. 資料處理引擎優化

| 面向 | 說明 |
|------|------|
| CSV/Excel 解析 | 動態欄位匹配，相容多種 Google Ads 報表格式 |
| 資料合併 | 多來源數據依據合併鍵（merge key）自動聚合 |
| 衍生欄位 | 支援欄位間運算（如 CTR = 點擊/曝光） |
| 列過濾 | 依條件篩選原始資料列 |
| 軟刪除 | 原始記錄與處理記錄支援軟刪除與復原 |

---

## 三、變更影響評估

| 面向 | 影響 | 因應措施 |
|------|------|----------|
| 數據遷移 | 現有資料需歸屬至專案 | 透過 Seeder 建立範例專案並遷移資料 |
| 使用者體驗 | 登入後需選擇專案 | 設計直覺的專案選擇介面 |
| 權限管理 | 角色體系更複雜 | 提供清楚的權限說明與預設角色 |
| 維護成本 | 多租戶架構增加查詢複雜度 | 優化查詢邏輯，使用子查詢避免關聯載入問題 |
